

## Subtitle font rendering (root cause + fix) — VERIFIED with real renders
- **Bug:** chosen subtitle font (e.g. Bebas Neue) did NOT render in final MP4 — libass fell back to default sans. Color/stroke/shadow worked (ASS override tags) but font family didn't.
- **Root cause (proven):** remote FFmpeg workers don't have Google fonts installed, AND the FFmpeg API BLOCKS the `fontsdir` option (any command with `fontsdir` → "Bad Request"). So `subtitles` filter couldn't locate the custom font.
- **Fix (in `lib/captions/font-embed.ts` + `lib/compositor.ts`):** embed the chosen `.ttf` as an MKV attachment; libass auto-loads embedded fonts. Pipeline: Step A = mux `.ass` + `.ttf` into tiny `.mkv` (stream copy) via FFmpeg API; **re-host that MKV to S3 via uploadPublicBuffer** (CRITICAL: cdn.abacus.ai FFmpeg-output URLs are UNREACHABLE as inputs to subsequent FFmpeg jobs — only S3 / fal.media URLs are fetchable as inputs); Step B = composite with `subtitles={{in_subs}}`. Graceful fallback to `{{in_ass}}` if embed fails.
- 15 bundled font files live in `public/fonts/subtitles/*.ttf` (family names match SUBTITLE_FONTS). `fontFileFor(family)` maps family→file.
- **WYSIWYG preview:** `app/globals.css` has 15 `@font-face` rules pointing to the SAME `/fonts/subtitles/*.ttf` files, so editor + reel-player preview use the exact fonts the server embeds. `components/reel-player.tsx` overlay now honors `subtitleStyle` (fontFamily/color/stroke/shadow/position/highlight) — pass `subtitleStyle={reel?.subtitleStyle}` from reel page.
- Variable fonts render bold correctly with ASS `Bold=-1` (verified, not Thin).
